Verifiable by design

Trustless storage you can check yourself

Every file in WalDrive is backed by two public facts — a content-addressed blob on Walrus and a metadata object on Sui. The app just makes them fluid.

01

Content-addressed

The Walrus blob ID is the cryptographic fingerprint of your bytes. Same content, same ID — tampering is mathematically loud.

02

You own the storage

Uploads mint an on-chain Blob object sent to your wallet, plus a FileRecord with name, tags, versions and expiry — signed by your local key, in-process, no popup.

03

Anyone can verify

Hit Verify and WalDrive re-fetches the bytes from a public aggregator and hashes them on screen. No wallet needed — share the link and the recipient can check too.

Architecture

No backend. Two networks.

Metadata on Sui Move objects

  • FileRecord — name, mime, size, tags, version chain, expiry
  • Folder — nested folders as owned objects
  • Signed in-process by your local Ed25519 key
  • Browse, rename, tag, trash — every action is a transaction

Bytes on Walrus erasure-coded

  • Files erasure-coded across storage nodes
  • Blob ID = content commitment (Merkle root)
  • Public aggregator reads — no wallet, no gate
  • Storage object owned by you: renew or delete
For agents

Your agents write. You browse.

WalDrive ships an MCP server — the agent write-path. AI clients and scripts store artifacts and memory to the same Walrus data you manage in the console: upload_file list_files download_file get_file_info

Humans get the fluid console; agents get a clean protocol. Same chain, same blobs, same truth.
